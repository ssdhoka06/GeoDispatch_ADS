01 Project Overview — Non-Technical
What is GeoDispatch?
GeoDispatch is an emergency facility dispatch optimizer. Given a city map with hospitals, fire stations, or ambulance
depots plotted on it, the system answers two questions in real time: (1) which facility is closest to an incoming
emergency incident, and (2) how well does the current distribution of facilities actually cover the city — and where are
the gaps?
The system is interactive. A user can click the map to add a new facility, click an existing facility to mark it offline
(overloaded or under maintenance), and immediately see the coverage zones redraw. It also runs an optimisation
that suggests where a new facility should be placed to maximally reduce uncovered area.
Why does this matter in the real world?
This exact problem is solved — in various forms — by Uber (nearest driver), Google Maps (nearest pharmacy),
national EMS services (ambulance base placement), and city planners (hospital coverage analysis). The algorithms
underpinning GeoDispatch are not toy examples — they are the actual algorithms used in spatial databases and
geographic information systems.
What makes this academically significant?
The project is built on a mathematical duality: every Voronoi cell for a facility site is precisely the set of all geographic
points whose nearest-neighbor in the KD-tree is that facility. This means the two data structures are not independent
— they describe the same spatial partition from two different perspectives. Exploiting this duality allows the system to
perform incremental updates (when a facility goes offline, only the affected neighboring cells are recomputed) rather
than rebuilding the entire diagram. This reduces worst-case cost from O(n log n) to O(k log n), where k is the number
of neighboring cells affected.

KD-Tree Indexing
Viva story: "I built the spatial index that powers every single dispatch query in this system."
P1 owns the foundational KD-tree module. Everything in this project — nearest-facility lookup, k-NN queries,
incremental update neighbour detection — depends on P1's code being correct and fast. P1 writes kd.h and kd.c. P2
will extend kd.h with dynamic operations in a separate file (kd_dynamic.c) without modifying P1's files. P1 must

C
Task 1 — KD-tree construction — kd_build()
Objective: Build a static balanced KD-tree from an array of 2D points in O(n log n) time.
Data structures to define in kd.h:
typedef struct { double x; double y; int id; } point_t;
typedef struct kdnode {
point_t point;
int axis; // 0 = split on x, 1 = split on y
int deleted; // P2 will use this flag
struct kdnode* left;
struct kdnode* right;
} kdnode_t;
Implementation notes:
■ Use a slab (pool) allocator: allocate n kdnode_t structs in one malloc call at the start of kd_build.
Store a pointer to the slab in a global or passed-in context. This avoids n separate malloc calls.
■ Partition using a two-pointer quickselect to find the median element in-place on the current subarray.
Do NOT sort the full array — that would be O(n log^2 n). Quickselect gives O(n) expected per level.
■ Alternate axis: depth % 2 == 0 splits on x, depth % 2 == 1 splits on y.
■ The median element becomes the node. Left subtree receives indices [lo, mid-1], right receives
[mid+1, hi].
■ Set the deleted flag to 0 on every node during construction. P2 will set it to 1 during soft-delete.
Function signature:
kdnode_t* kd_build(point_t* pts, int n);
Testing: build with 10 known points, print the tree with an in-order traversal, verify the split axes alternate correctly.

C
Task 2 — Nearest-neighbor query — kd_nearest()
Objective: Given query point q, return the closest non-deleted point in the tree.
Algorithm — branch-and-bound:
■ Maintain best_dist (infinity initially) and best_node (NULL).
■ At each node: if node is deleted, skip distance check but still recurse. Otherwise update best if
dist(node, q) < best_dist.
■ Determine which child subtree q falls into (near side). Recurse into near side first.
■ After returning from near side: check if the far side hyperplane is within best_dist. If |q.axis -
node.point.axis| < best_dist, recurse into far side too. Otherwise prune it.
■ This pruning is the bounding hypersphere check — the key to O(log n) average performance.
Function signature:
point_t kd_nearest(kdnode_t* root, point_t q);
Edge case: if all nodes are deleted, return a point_t with id = -1 as a sentinel.
C
Task 3 — k-NN query — kd_knn()
Objective: Return the k closest non-deleted points to query q, sorted nearest-first.
Data structure: max-heap of size k, implemented inline in C.
■ Define a heap_entry_t struct: { double dist; point_t pt; }
■ Maintain a fixed-size array heap[k] and an integer heap_size.
■ heap_push: if heap_size < k, insert and sift up. If heap_size == k and new dist < heap[0].dist, replace
root and sift down. Reject otherwise.
■ After traversal, the heap contains the k nearest. Sort descending by distance before returning (since
the heap is a max-heap, index 0 is the farthest of the k nearest — reverse it for the caller).
■ Pruning: during tree traversal, only explore a subtree if its closest possible point is nearer than the
current kth-nearest (heap[0].dist).
Function signature:
point_t* kd_knn(kdnode_t* root, point_t q, int k, int* out_count);
Caller is responsible for freeing the returned array. out_count will be <= k (may be less if fewer than k non-deleted
nodes exist).

C
Task 1 — Soft-delete and dead-ratio tracking — kd_delete()
Objective: Mark a point as deleted without restructuring the tree. Track how many nodes are dead.
Implementation in kd_dynamic.c:
■ Traverse the tree as in kd_nearest to locate the node with matching id.
■ Set node->deleted = 1. Do NOT free the node or relink pointers.
■ Maintain two global (or context) counters: dead_count and total_count. Increment dead_count on
delete.
■ kd_dead_ratio() returns (double)dead_count / total_count.
■ kd_nearest() and kd_knn() (written by P1) already skip deleted nodes because they check the
deleted flag before updating best.
Function signatures:
void kd_delete(kdnode_t* root, int point_id);
double kd_dead_ratio(kdnode_t* root);
Do not touch P1's kd.c. Only add to kd.h and write in kd_dynamic.c.
C
Task 2 — Lazy rebalance and online insert — kd_rebalance() and kd_insert()
Objective: Support adding facilities at runtime and rebuilding the tree when it degrades.
kd_insert():
■ Traverse the tree following the same axis-split rules as construction.
■ Insert as a new leaf node (allocate from the pool if space remains, else a single malloc).
■ Increment total_count. After insert, check if dead_ratio > 0.30 or tree height exceeds 2*log2(n). If so,
call kd_rebalance() automatically.
kd_rebalance():
■ Collect all non-deleted nodes into a flat array (in-order traversal).
■ Free the old slab. Call kd_build() on the collected array. Replace the root pointer.
■ Reset dead_count = 0. total_count = number of live nodes.
Function signatures:
void kd_insert(kdnode_t** root, point_t p);
void kd_rebalance(kdnode_t** root);

P3 owns voronoi.h and voronoi.c. This is the most algorithmically demanding single module in the project. P3 does

C
Task 1 — Fortune's sweep line algorithm — voronoi_build()
Objective: Construct the complete Voronoi diagram of n sites in O(n log n) using Fortune's algorithm.
Key data structures needed inside voronoi.c (not exposed in header):
■ event_t: { double x; double y; int type; /* SITE or CIRCLE */ int site_id; arc_t* arc; }
■ event_queue: a min-heap of event_t sorted by x (sweep line moves left to right).
■ beach_line: a balanced BST (you can use a simple sorted linked list for correctness first, then
optimise) of arc_t nodes, each representing a parabolic arc.
■ arc_t: { int site_id; arc_t* prev; arc_t* next; event_t* circle_event; half_edge_t* edge; }
Algorithm phases:
■ Phase 1 — Initialise: insert all n site events into the priority queue (sorted by x-coordinate). Build the
DCEL skeleton with one face per site.
■ Phase 2 — Process events: while queue is not empty, pop the minimum-x event. If SITE event: find
the arc directly above the new site on the beach line (binary search by y). Split that arc into three arcs.
Create new half-edges in the DCEL. Check for new circle events involving the new arc's neighbours
and add valid ones to queue. If CIRCLE event: remove the collapsing arc from the beach line. Finalise
the Voronoi vertex in the DCEL. Add new circle events for the newly adjacent arcs.
■ Phase 3 — Finalise: complete any remaining infinite edges by extending them to the bounding box
(P4 will clip them properly with Sutherland-Hodgman, so P3 just needs to mark them as infinite).
Function signature:
dcel_t* voronoi_build(point_t* sites, int n);
Start with a correct O(n^2) implementation if Fortune's is too complex initially. Correctness first, then optimise to O(n log
n). The professor will ask about the algorithm, not necessarily the implementation details.

Task 3 — Incremental site insertion — voronoi_insert_site()
Objective: Add a new facility to the existing Voronoi diagram without full rebuild.
■ Locate which face the new site falls into by doing a point-in-polygon walk on the DCEL (start from any
face, check if new site is inside using winding number, step to adjacent face via twin if not).
■ That face is the one being split. Extract its boundary polygon.
■ Run a local Fortune sweep considering only the new site and the boundary sites of that face plus its
immediate DCEL neighbours.
■ Update the DCEL: remove the split face, insert new faces and edges for the new site and the
modified neighbours.
Function signature:
void voronoi_insert_site(dcel_t* d, point_t new_site);
This function is used by P5's algo.c for the incremental update. P3 must ensure the DCEL remains valid (all twin
pointers consistent) after insertion.

Viva story: "I solved the geometry problems — how to bound, measure, and classify every Voronoi cell."
P4 owns geometry.c, which links against voronoi.o and operates on the DCEL produced by P3. P4 never modifies

C
Task 1 — Sutherland-Hodgman polygon clipping — clip_to_bbox()
Objective: Fortune's algorithm produces infinite half-edges. Clip them to the bounding box of the dataset so
every cell becomes a valid closed polygon.
Algorithm — Sutherland-Hodgman:
■ The clipping region is the bounding box: xmin, ymin, xmax, ymax (from P2's pune_meta.json,
expanded by 10%).
■ For each face in the DCEL: collect its vertex ring by walking the outer_edge chain.
■ Apply Sutherland-Hodgman clipping against each of the 4 clip edges (left, right, bottom, top) in
sequence.
■ For each clip edge: iterate over output polygon from previous step. For each pair of consecutive
vertices (A, B): if A inside and B inside: emit B. If A inside and B outside: emit intersection. If A outside
and B inside: emit intersection then B. If both outside: emit nothing.
■ Replace the face's half-edge ring in the DCEL with the clipped polygon vertices. Mark all edges as
non-infinite.
Function signature:
void clip_to_bbox(dcel_t* d, double xmin, double ymin, double xmax, double ymax);
After this function runs, all DCEL faces are closed bounded polygons. P4 calls this once after voronoi_build().
C
Task 2 — Cell area via Shoelace formula — cell_area()
Objective: Compute the area of each Voronoi cell and store it on the face_t struct.
Shoelace formula for a polygon with vertices (x0,y0)...(xn-1,yn-1):
■ area = 0.5 * abs( sum over i of (x_i * y_{i+1} - x_{i+1} * y_i) ) where indices wrap around.
■ Walk the face's half-edge ring using next pointers. At each half_edge, record its origin vertex (x, y).
■ After collecting all vertices, apply the Shoelace formula.
■ Store the result in face->area (the field is already defined in face_t by P3).
■ Units are square metres (since P2's projection gives Cartesian coords in metres).
Function signature:
double cell_area(dcel_t* d, int face_id);
Also write a loop that calls cell_area() for every face and populates all face->area fields. Call this after clip_to_bbox().

Task 1 — Incremental Voronoi update — voronoi_incremental_update() in algo.c
Objective: When a facility changes state, update only the affected Voronoi cells rather than rebuilding the
entire diagram. This is the O(k log n) algorithm that is the project's research contribution.
This function is the embodiment of the KD-tree / Voronoi duality.
■ Step 1: Given the site_id of the changed facility, call dcel_neighbours(d, site_id, &k;) to get its k
adjacent faces. This uses the DCEL — O(k) where k is the cell degree (typically 5-7 for Voronoi
diagrams).
■ Step 2: For each of those k neighbour face site_ids, call kd_knn(root, neighbour_site, k+2) to find the
nearby facilities that might now be the nearest neighbour of points in that zone. This uses the KD-tree
— O(k log n).
■ Step 3: Collect the union of the changed site, its neighbours, and the KD-tree candidates. Run
voronoi_build() on this local subset only (typically 10-15 sites). Splice the result back into the main
DCEL.
■ Step 4: Recompute cell_area() and flag_underserved() for only the affected faces.
Function signature:
void voronoi_incremental_update(dcel_t* d, kdnode_t* kd_root, int changed_site_id);
Benchmark this: log the number of affected cells k and the wall-clock time. Compare against a full voronoi_build() call.
This is the data that proves O(k log n) empirically.

Task 2 — Lloyd's algorithm — run_lloyds() in algo.c
Objective: Iteratively move each facility to the centroid of its Voronoi cell to minimise maximum coverage
distance. This is the facility placement optimiser.
■ For each iteration up to max_iterations:
■ For each face in the DCEL: walk the half-edge ring to collect vertices. Compute centroid: cx = mean
of vertex x-coords, cy = mean of vertex y-coords.
■ Compute movement delta: sqrt((new_cx - old_x)^2 + (new_cy - old_y)^2). Accumulate total
movement across all facilities.
■ Move the facility: update the point_t in the KD-tree (soft-delete old, insert new). Call voronoi_build()
on the full updated set (Lloyd's always does full rebuilds).
■ If total_movement < convergence_threshold (e.g. 50.0 metres), break.
■ Record each iteration's state: for each facility, {id, from: {x,y}, to: {x,y}}.
Function signature:

13 Frontend Specification — Built Collectively After All C Tasks Complete
The frontend is built by all five team members together once the REST API is running. No individual owns it. The
design goal is a cinematic, high-fidelity spatial interface that feels nothing like a student project — it should look and
feel like a professional operations dashboard that could be deployed by an actual emergency services authority.
13.1 Visual Identity
The palette is deep space: near-black backgrounds (#0D0D1A), electric indigo accents (#6C63FF), signal teal
(#00E5CC), alert coral (#FF5C5C), and pure white text (#F0F0FF). Typography is a single variable font — Inter
Variable — set at multiple optical sizes. All interactive elements have a 2px glow on hover in the accent colour. The
aesthetic is between a military command centre and a high-end data journalism piece — precise, purposeful, and
arresting.
13.2 Layout Structure
The layout is a full-bleed single page with three layers stacked by z-index:
■ Layer 0 (z=0): The Three.js particle field — 4000 slow-drifting white particles on the deep background, visible in
the left panel and the top header area. Not on the map — only in UI chrome.
■ Layer 1 (z=1): The Leaflet map fills the right 65% of the viewport, edge to edge, no border, no padding. The map
tiles are a custom dark style (CartoDB Dark Matter or Stadia Alidade Smooth Dark) so our D3 overlays pop.
■ Layer 2 (z=2): The D3 SVG overlay is positioned absolutely on top of the Leaflet map, synced to the map's
coordinate system. Voronoi polygons, facility markers, and connection lines all live here.
■ Layer 3 (z=3): The glass-morphism left panel (35% width) — blurred backdrop, border 1px
rgba(108,99,255,0.3), stats cards, controls, search.
■ Layer 4 (z=4): Toast notifications, modal overlays, and loading screens.
13.3 The Map and Voronoi Layer — D3.js
The Voronoi polygons are drawn as SVG paths synced to the Leaflet map projection. When the map pans or zooms,
the D3 overlay recomputes the projected coordinates using Leaflet's latLngToLayerPoint() and redraws. The visual
treatment of the cells:
■ Resting state: fill is a radial gradient from the facility's accent colour at 8% opacity at the centroid to transparent
at the cell edges. Stroke is the accent colour at 40% opacity, 0.8px weight. The result is a glowing membrane
effect — cells breathe with a slow CSS pulse animation (opacity oscillates between 0.3 and 0.6 on a 3-second
ease-in-out loop, staggered per cell using cell index * 0.15s delay).
■ Underserved cells: stroke switches to coral (#FF5C5C), weight increases to 2px. The fill gradient shifts to coral.
A subtle scanline pattern (repeating linear gradient of 1px transparent / 3px rgba(255,92,92,0.05)) overlays the fill.
The cell pulses faster — 1.2 second loop.
■ Hover state: GSAP animates the hovered cell's stroke-width from 0.8 to 2.5px and opacity from 0.4 to 0.9 over
180ms ease-out. All neighbouring cells dim to 20% opacity simultaneously (GSAP timeline). A tooltip card appears
at cursor position.
■ Active (clicked) state: The cell expands by 3% scale (CSS transform on the SVG path) with a 250ms spring
ease. The left panel slides to show that facility's detail view.
